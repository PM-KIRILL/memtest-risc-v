Одной из целей при переводе MemTest86+ на мультиархитектурную основу (еще не достигнуто) является размещение всего платформозависимого кода внутри подкаталога `arch/<arch-name>/`.
В идеале в кроссплатформенной части кода (т. е. за пределами `arch/`) не должно быть никаких `#if defined(__i386__)` и тому подобных конструкций.

## Структура исходного кода

Поскольку исходный код MemTest86+ представляет собой смесь C и ассемблера, а некоторые заголовочные файлы включаются в оба типа файлов, следует соблюдать определенную осторожность.

Как правило, структура подкаталогов в `arch/` произвольна, за исключением некоторых файлов, которые должны присутствовать всегда:
* `config.mk` — файл, включаемый в основной `Makefile.arch`. Он должен содержать определения некоторых переменных, см. `Makefile.arch` и соседние файлы `config.mk`.
* `arch.h` — заголовочный файл, который включается практически в любой исходный файл на C.
* `defs.h` — содержит **определения**, безопасные для ассемблера. Ожидается, что они будут подключаться через `#include` как можно раньше.
* `reloc.inc.c` — фрагмент кода, включаемый в `reloc.c`.
* `test.inc.c` — включается внутрь `test.c` и должен содержать поддержку оптимизаций тестов (обычно просто ассемблерные листинги), если таковые имеются. Нет необходимости указывать `-O0` где-либо, так как это НЕ отдельная единица трансляции — файл компилируется как часть основного `test.c`.

Помимо этого, обычно должны существовать один или несколько файлов ассемблера (*.S) для правильной настройки контекста процессора и один или несколько исходных файлов C для высокоуровневой платформозависимой поддержки. Обычно они перечисляются в переменных `ASM_SOURCES` и `ARCH_OBJS` в файле `config.mk` (вероятно, вы захотите указать файлы ассемблера в правильном порядке в переменной `ASM_SOURCES`).

## Об отладке оптимизированных тестов

При восстановлении стандартных реализаций тестов на чистом C выяснилось, что некоторые из них неисправны. Хотя можно просто прописать `#define OPTIMIZED 0`, чтобы полностью отключить **все** оптимизированные фрагменты, иногда полезно отключить только первый, второй или третий фрагмент.

Более того, обратите внимание, что некоторые тесты устроены следующим образом:

    snippet1(); // Первоначальное заполнение памяти
	for (i=0; i<iter; i++) {
	    snippet2(); // проверка и изменение
		snippet3(); // проверка и восстановление
	}

`snippet1()` и `snippet3()` здесь должны оставлять память в идентичном состоянии, но первый цикл тестирования имеет уменьшенное количество итераций. Иногда `iter` оказывается равным 1, поэтому проблема становится заметной только на втором и последующих циклах.
